库之间的关联关系：完全打通

选任意一个客户/讲师/课程/项目/方案/投标，AI 实时把相关数据拉通，任何反向查询都成立：讲师能查“出现在哪些方案和投标”，项目能查“用了哪些老师和课程”。

核心表达：每个模块都不是孤岛，所有实体都是 N:N 关联

讲师、课程、项目、需求、方案、投标、客户反馈之间不是固定顺序。任意选中一个实体，都能反查它关联过的其他实体，并继续向外扩展。

标签与规则 · 一句话说清

这一页配的东西控制 AI 怎么找资料、怎么打分、什么时候停下来问业务。一句话：你这里改一改，下面 7 个地方立刻按新规则跑。

业务规则 · 不是技术参数
标签 给资料打业务标记

例：金融 / 中高层 / 银行 / 对客可引用

→ 决定 AI 搜索时能不能找到、SKILL 能不能匹配到正确资料

⚠ 标签错 = 找不到 / 推错

规则 业务底线 + 判断口径

例：客户曾拒绝该讲师 → 一票否决；资料过期 → 扣分

→ 决定候选能不能进入、要不要扣分、要不要提示风险

⚠ 规则缺 = AI 生成的看似可用，实际不可信

权重 各维度的重要性

例：需求标签 25% / 课程 20% / 历史项目 20% / 讲师能力 20%

→ 决定候选排序时哪个因素分量更重

⚠ 权重不对 = 排序偏（推像但不准的人）

追问条件 什么时候停下来问业务

例：缺客户战略 / 缺反馈 → AI 不继续硬做，先反问

→ 决定 AI 优先准确还是优先速度

⚠ 太松乱生成；太紧频繁卡

5 种存储分别存什么 · 查什么

单一存储不够：图 DB 强在关系遍历但弱在全文检索；向量库强在语义但弱在精确匹配。每种存储都有不可替代的场景。

🕸️
图数据库 NebulaGraph / Neo4j
存什么
实体（讲师/课程/客户/项目/方案/反馈）+ 关系（讲过/被使用/合作过/反馈）
查什么
"完全打通"360° / "王老师出现在哪些方案" / "金融领导力被哪些客户用过" / 多跳关系遍历
不可替代

🧠
向量数据库 Qdrant / Milvus / pgvector

不可替代

部署拓扑建议（按规模）

从单机演示到大规模集群的演进路径，不要一开始就上重架构。

🧪 单机演示（团队内试用）

Docker Compose 一把起；PG（含 pgvector）+ Qdrant + ES + MinIO + NebulaGraph 单节点。

资源：8C / 32G / 200G SSD · 单机 OK

数据规模：≤ 1 万份资料 · 10 人内

🏢 部门级（30–100 人售前团队）

每个组件独立 VM（3 节点起）；图 DB / ES / PG 都做主备；MinIO 做 4 节点擦除码。

资源：约 5 台 16C / 64G + 10 TB 存储

数据规模：≤ 50 万份资料 · 100 人

🌐 集团级（多业务线）

K8s 集群部署 / NebulaGraph 分片 / ES 集群 / PG 主从 + 读写分离 / MinIO 分布式。检索层加 rerank 模型独立部署。

资源：20+ 节点 · GPU 推理节点

数据规模：百万级 · 千人
