  ▎ "今天主要聊 RAG —— 检索增强生成。一句话讲就是：让大模型在回答你之前，先去查一遍资料，然后基于查到的资
  ▎ 料回答，而不是凭它训练时学过的知识乱讲。"
  ▎
  ▎ "做 Agent 的时候，RAG 做得好不好，直接决定 Agent 的业务上限。所以我们梳理了一套全链路实战指南。"

  为什么开场这样讲：
  - "开卷考试" 这个比喻立刻让对方对得上
  - "决定 Agent 上限"这句话把 RAG 提到战略层面，比"是个技术"重得多

  二、主线 5 分钟讲清楚 RAG 全链路
  
  Step 1: 文档加载 —— "把各种格式的资料变成 AI 能读的文本"

  ▎ "PDF、Word、PPT、Excel 都要能解析，扫描件还要走 
  ▎ OCR。这一步关键不是技术难，而是要把文档的元数据也提取出来 —— 
  ▎ 文件名、章节、页码、作者，后面检索的时候这些元数据能救你的命。"
  
  金句："加载是 RAG 的入口，入口质量决定后面一切。"

  Step 2: 切分（Chunking） —— "把长文档切成 AI 一次能消化的小块"

  ▎ "为什么要切？因为大模型每次只能塞固定长度的内容。但切的时候不能瞎切 —— 切错了句子 / 切错了段落，AI 
  ▎ 看到的就是断章取义。"
  ▎
  ▎ "业界现在比较前沿的是 Meta-Chunking，用大模型自己的"困惑度"判断"在哪里切最合理" —— 
  ▎ 简单说就是：模型看一段文本时，如果突然"困惑度飙升"，说明这里逻辑断了，就在这里切。"

  金句："切分不是按字数 / 按段落，而是按语义边界。"

  Step 3: 索引（Embedding） —— "把文本变成向量，让机器能算"相似度""

  ▎ "这一步技术细节挺重，简单讲：每段文本通过 embedding 模型变成一串数字（向量）。两段文本意思越接近，向量
  ▎ 之间的"距离"越短。所有文档都向量化后存进向量数据库，将来检索时按距离找最近的。"
  ▎
  ▎ "模型选型很重要 —— 千问、BGE、OpenAI text-embedding 各有强项，中文场景一般用 BGE 或千问。"

  金句："embedding 决定了你的检索"看不看得懂中文 / 看不看得懂行业术语"。"

  Step 4: 检索（Retrieval）—— 这一步最值得讲，水最深

  ▎ "用户提问 → 问题向量化 → 拿这个向量去库里找最近的 N 
  ▎ 个文档片段。但直接拿用户问题去搜，效果往往很差，因为 —— "

  3 个进阶技术（任选 1-2 个讲透）：

  (a) Query 改写 —— 多轮对话和口语化处理

  ▎ "用户问'它多少钱'，'它'指谁？前一轮聊的华为 Mate 60。RAG 系统得先把'它'还原成具体实体，否则查不到。"

  (b) HyDE（假设文档） —— 业内当红技术

  ▎ "这个有意思 —— 用户问'为什么面团会发起来'，知识库里的真实答案是论文里写的'酵母代谢 + 二氧化碳释放 + 
  ▎ 面筋蛋白网络'。两者的向量距离其实很远，因为用户用口语，文档用术语。
  ▎
  ▎ HyDE 的做法：先让 AI 瞎编一个假答案（凭它训练时的知识），然后用'假答案的向量'去搜真实文档。因为假答案
  ▎ 的措辞跟真实文档接近，命中率反而提升一大截。"

  金句："用问题搜答案，不如用'假答案'搜真答案。这是非对称检索的破解之道。"

  (c) 重排序（ReRank） —— 召回多 → 精排少

  ▎ "检索召回阶段是'粗筛'，先拿 50 条出来。然后再用一个专门的 ReRank 模型对这 50 条做精排，挑出最相关的 5 
  ▎ 条给大模型。
  ▎
  ▎ 为什么不一次性精确召回？因为精确召回慢，召回先广撒网，再精排，效率最高。"

  Step 5: 生成（Generation） —— 把检索到的资料 + 问题打包给大模型

  ▎ "这一步是大模型用'开卷考试'方式给答案。看起来简单，坑很多："
  ▎
  ▎ - 模型可能忽略提供的资料，用它训练时的旧知识答题
  ▎ - 检索回来 5 个片段里有 2 个互相矛盾，模型困惑
  ▎ - 长上下文里中间位置的关键信息被忽略（"lost in the middle"）
  ▎
  ▎ "对应的优化方式：prompt 里强制要求'引用资料 + 标注来源'、用专门的 SFT 微调、参数调控（temperature / 
  ▎ top_p）。"

  三、压轴的部分：评测体系（一定要讲，对面最容易缺这块）

  framing：

  ▎ "前面讲的所有优化，没有评测都是空谈 —— 你怎么证明你的 RAG 比上周的好？怎么向老板汇报 
  ▎ ROI？怎么知道某个改动是修复了 bug 还是引入了新 bug？"

  Ragas 框架 5 个核心指标，用人话讲：

  ┌───────────────────┬────────────────────────────┬────────────────────────┐
  │       指标        │          人话翻译          │      答不好会怎样      │
  ├───────────────────┼────────────────────────────┼────────────────────────┤
  │ Context Precision │ 找回来的资料相关吗         │ 资料看着像但答非所问   │
  ├───────────────────┼────────────────────────────┼────────────────────────┤
  │ Context Recall    │ 该找的资料漏了吗           │ 答案不完整、缺关键依据 │
  ├───────────────────┼────────────────────────────┼────────────────────────┤
  │ Faithfulness      │ 答案是不是基于资料         │ AI 在瞎编 / 幻觉       │
  ├───────────────────┼────────────────────────────┼────────────────────────┤
  │ Answer Relevancy  │ 答案跟问题对不对得上       │ 答非所问               │
  ├───────────────────┼────────────────────────────┼────────────────────────┤
  │ Noise Sensitivity │ 资料里掺了干扰还能不能答对 │ 容易被噪音带偏         │
  └───────────────────┴────────────────────────────┴────────────────────────┘

  金句："Ragas 用 LLM 来评估 LLM —— 自动化、可量化、贴近人工评测的精度。"

  评测集自动生成：

  ▎ "以前评测要人工标注 100 道题，太贵。Ragas 能自动从知识图谱里生成测试集，覆盖单跳 / 多跳、具体 / 抽象 4
  ▎  种场景。"

  3. "重排序是大多数团队漏掉的一步，但它是从能用到好用的关键跳。"

  五、Q&A 防被问倒
  
  Q1: "RAG 跟搜索引擎有啥区别？"

  ▎ "搜索引擎找网页，按关键词匹配 + PageRank。RAG 
  ▎ 找语义片段，按向量相似度，而且最后还要让大模型基于这些片段生成自然语言回答。RAG 是'搜索 + 
  ▎ 生成'的合成体。"
  
  Q2: "为什么不直接 fine-tune 模型？"

  ▎ "三个原因：(1) fine-tune 贵且慢，迭代周期长；(2) 模型一旦 fine-tune 
  ▎ 了，知识就固化了，文档一更新又要重训；(3) RAG 可追溯，每个答案都能指回到具体文档，fine-tune 
  ▎ 之后模型说的话你不知道从哪来的。"
  
  Q3: "为什么 RAG 还会有幻觉？"

  ▎ "RAG 不能消除幻觉，只能降低。常见原因：检索没找对资料（系统局限）、找到了但 prompt 
  ▎ 没强约束（让模型用资料而不是瞎编）、长上下文导致中间信息丢失。所以评测体系里要看 Faithfulness 
  ▎ 这个指标专门量化幻觉。"
  
  Q4: "Graph RAG 跟普通 RAG 怎么选？"

  ▎ "普通 RAG 已经能 cover 80% 场景。Graph RAG 主要解决两类：多跳推理（A 影响了 B，B 发明了 C，问谁影响了 
  ▎ C 的发明者）和全局摘要（这本小说的主旨是什么）。如果你的业务大量涉及实体之间的关系，可以上 Graph 
  ▎ RAG；如果只是文档问答，普通 RAG 性价比更高。"
  
  Q5: "做 RAG 需要多少资源？周期多长？"

  ▎ "看复杂度。MVP 版本两个工程师 1-2 周能跑通：数据接入 + embedding + 简单检索 + 调通 
  ▎ prompt。但做到可交付级别需要 2-3 个月，主要花在：评测体系搭建、Query 改写策略、ReRank 集成、Graph 
  ▎ 关系建模、Bad case 迭代。"
