1. 概述

本文档定义 售前 AI 工作台 的管理后台需求。 工作台前端是 Forge 桌面端（已开发，对话式 AI + Skill 编排）， 本管理后台负责对支撑前端的数据 / 配置 / 监控进行运营管理， 是后台运营 / 产品 / 业务管理员使用的 Web 系统。

管理后台覆盖：6 个知识库 · 6 个 SKILL · 5 个外部系统集成 · 用户权限 · 数据看板。 不包括前端售前用户的 chat 界面（那是 Forge 桌面端的职责）。

本文档解决三个问题：
  1. 管理后台有哪些模块、每个模块做什么？
  2. 每个模块的关键页面 / 字段 / 操作是什么？
  3. 开发优先级、依赖、风险？

边界与不包含

  • ✅ 包含：知识库 CRUD、SKILL 编辑、文档解析流水线、系统集成配置、用户权限、运营看板
  • ❌ 不包含：售前用户的 chat 界面（Forge 桌面端已交付）
  • ❌ 不包含：客户面向的方案展示门户（独立项目）
  • ❌ 不包含：投标流程（场景 5/6，第二阶段）

M2 · 知识库管理 P0

2.1 概述

6 个知识库共用一套 CRUD + 上传 + 标注框架，差异在元数据 schema 和审核规则。

KB规模主数据源是否需要人工审核审核难度
KB-1 讲师库内部 30-50 + 51CTO 2000+讲师后台 API + docx 导入 + 51CTO API否（API 是权威）—
KB-2 课程库100-200 + 51CTO 900+xlsx 导入 + 51CTO API部分（dimension 标签）低
KB-3 方案库100-300（精选）pptx/docx 批量上传是（必审核）高
KB-4 客户库已有 + 持续CRM API 同步否（CRM 是权威）—
KB-5 话术库200-500 段业务专家手动录入是（法务审）中
KB-6 行业库6-10 行业 × 30-50 段季度运营整理是（数据团队）低

2.2 通用功能（每个 KB 都有）

  • 列表页：筛选 + 搜索 + 批量操作（导出 / 删除 / 重打标签）
  • 详情页：单条记录 / 元数据 / 内容片段（chunks）/ 引用历史
  • 编辑页：手动编辑元数据 + chunk 内容
  • 新增页：批量上传（M4 流水线）/ 单条录入
  • 引用追踪：这条数据被哪些 SKILL 在何时引用过
  • 版本历史：每次编辑记录 diff + 谁改的
  • 下架 / 上架：临时禁用某条不删除（如"讲师暂停合作"）

2.3 KB-1 讲师库

元数据 schema

{
  id, name, alias?,             // 真名 / 化名
  affiliation,                  // 单位
  title,                        // 职称
  category: "学院"|"产业"|"hybrid",
  is_academician: boolean,
  expertise: string[],           // 能力标签
  capability_dimensions: string[],
  price_per_day: number,
  availability_dates: Date[],   // 不可用日期
  cooperate_count: int,
  score: float,                  // 1-5
  customer_acceptance: { [industry]: avg_score },
  profile_text,                  // 履历全文
  source: "internal" | "51cto" | "manual",
  status: "active" | "paused" | "archived"
}

关键页面

  • 讲师列表：筛选条 = 类别 / 维度 / 价格区间 / 状态。每行：头像、姓名、类别 chip、维度 tags、价格、评分
  • 讲师详情：左侧基础信息卡 + 右侧 5 个 tab（履历 / 课程 / 客户反馈 / 引用历史 / 编辑日志）
  • 同步管理：手动触发 / 自动 cron 拉讲师后台 + 51CTO；展示上次同步时间、新增 / 更新数
  • 能力维度词典管理：可编辑维度词典 + 重新打标签（背景跑 LLM）

3.3 评估与黄金集

  • 黄金集管理：每个 SKILL 至少 30 个测试 case（input → 期望 output 关键点）
  • 自动回归：SKILL 每次保存触发黄金集回测，记录通过率
  • 错例库：业务用户标"AI 答错了"会进错例库 → 下版 SKILL 的输入
  • A/B 测试：v3 vs v4 灰度放量（10% / 30% / 50%），自动对比满意度

4.3 批量重跑（重要）

  • 后台异步跑 + 进度可见
  • 对比"旧 vs 新"标签 → 业务人 spot check

8. 里程碑

Phase 4 二阶段第 16-24 周 方案输出 / 内容包装 / 投标场景 全公司售前 + 投标使用

9. 风险登记

风险影响概率缓解
业务专家时间不到位 高 项目停滞 中 SLA 写进 KPI；前 2 周建立节奏
CRM API 拿不到 / 不开放 中 客户库只能 mock 高 Phase 1 用 mock；并行推动 IT 开 API
方案库审核工作量超预期 中 影响 KB-3 上线 高 先做 100 份精选；后续按需扩
合规禁止数据出境 高 不能用 OpenAI 高 双 provider 架构；国内合规模型保底
SKILL 输出质量不达预期 中 业务弃用 中 评估体系 + 黄金集 + 错例迭代
讲师数据敏感（真名问题） 中 法务卡上线 中 真名 / 化名映射 + 权限分级
预算超支（LLM token 费用） 低 但要监控 低 预算上限 + 模型分层 + 缓存
